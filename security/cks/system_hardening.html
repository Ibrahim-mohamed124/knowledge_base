Use SECCOMP to restrict system calls
  • Seccomp is a Linux kernel feature to restrict which system calls a process can make
  • To check if secomp is supported by the Linux kernel
grep -i secomp /boot/config-$(uname -r)
CONFIG_SECOMP=y

 

  • Seccomp work in different modes
    • mode 0 = disabled
    • mode 1 =  strict > all sys calls are denied except write, read, exit, and sigreturn functions  (To return from a signal handler (essential for the process to handle stack frames after a signal).
    • mode 2  = Filtered ( use the docker seccomp built in profiles)
      • examples

Pod security 

PodSecurityPolicy serves as an admission controller within Kubernetes, enabling cluster administrators to effectively manage security-related aspects of the Pod specification.

  • there are two policies:
    • pod security standard
      • Three profiles: restricted, privileged, baseline
    • pod security admission: enabled by default admission controller (PodSecurity)
      • pre-configured profiles that restrict the pods security contexts
      • Three modes: enforce, audit, warn
      • To enforce the PSA profile in an namespace
kubectl label namespace  waves pod-security.kubernetes.io/enforce=restricted

 

kubectl run test --image ubuntu --privileged -n waves
Error from server (Forbidden): pods "test" is forbidden: violates PodSecurity "restricted:latest": privileged (container "test" must not set securityContext.privileged=true), allowPrivilegeEscalation != false (container "test" must set securityContext.allowPrivilegeEscalation=false), unrestricted capabilities (container "test" must set securityContext.capabilities.drop=["ALL"]), runAsNonRoot != true (pod or container "test" must set securityContext.runAsNonRoot=true), seccompProfile (pod or container "test" must set securityContext.seccompProfile.type to "RuntimeDefault" or "Localhost")

PSA is an integrated solution within Kubernetes that offers built-in capabilities for governing pod security. The solution incorporates a webhook project designed for admission controllers, which ensures enforcement of the controls outlined in the Pod Security Standards (PSS). This admission controller approach bears similarity to the functioning of PAC (Policy as Code) systems.

To create hard coded Rego policies: Plain OPA  + kube-mgmt

  • Can be installed via helm chart here
  • Policies can be loaded into OPA dynamically via ConfigMap objects using the kube-mgmt sidecar container. The kube-mgmt sidecar container can also load any other Kubernetes object into OPA as JSON under data.
  • kube-mgmt assumes a ConfigMap contains policy or JSON data if the ConfigMap is:
    • Created in a namespace listed in the --namespaces option. If you specify --namespaces=* then kube-mgmt will look for policies in ALL namespaces.
    • Labelled with openpolicyagent.org/policy=rego for policies
    • Labelled with openpolicyagent.org/data=opa for JSON data
  • When a ConfigMap has been successfully loaded into OPA, the openpolicyagent.org/kube-mgmt-status annotation is set to {"status": "ok"}.

Installing Gatekeeper:

  • Install the CRDs and operator
kubectl apply -f https://raw.githubusercontent.com/open-policy-agent/gatekeeper/v3.22.0/deploy/gatekeeper.yaml

 

  • create  Constraint template
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate                    
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels                                 
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |                                     
        package k8srequiredlabels

        violation[{"msg": msg, "details": {"missing_labels": missing}}] {
          provided := {label | input.review.object.metadata.labels[label]}
          required := {label | label := input.parameters.labels[_]}
          missing := required - provided
          count(missing) > 0
          msg := sprintf("you must provide labels: %v", [missing])
        }

Container Sandboxing
  • Sandboxing: are  techniques that is used to isolate processes in the system
    • like: 
      • seccomp, AppArmor, SELinux, … (white listing and black listing) 
      • linux namespaces PID, and others

Other techniques to sandbox containers:

  • gVisor: is an additional layer between the container and the kernel to improve the separation between the containers and between container and the host 
  • Instead asking the kernel to to syscalls directly, the container talks with gvisor 

gVisor:
