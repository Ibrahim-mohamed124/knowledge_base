Kubernetes security can be implemented on four levels:

  1. In development, by integrating security testing early in continuous integration pipelines. (code)
  2. In distribution, by ensuring that the software supply chain is safe by verifying the integrity of workload components. (containers and registry)
  3. At deployment time, by validating workload attributes such as security parameters, and by using approved container images. (container, cluster)
  4. At runtime, by monitoring all elements of container runtime: the host operating system, network, storage, and container runtime; and ultimately restricting container activity to only sanctioned processes that are prevented from accessing unauthorized resources and from engaging in disallowed network communication.( container, cluster, cloud)

Red Hat Advanced Cluster Security Functions:

  1. Integration into CI/CD pipelines and image registries, RHACS provides continuous scanning and assurance of containers.
  2. RHACS continuously scans the secured environment against Center for Internet Security (CIS) benchmarks and other security best practices, and prevents misconfigurations and threats, to deliver comprehensive Kubernetes-native protection.
  3. RHACS monitors, collects, and evaluates system-level events such as process execution, network connections and flows, and privilege escalation within each container in secured Kubernetes environments. Combined with behavioral baselining and allowlisting, RHACS detects anomalous activity that might indicate malicious intent, such as active malware, unauthorized credential access, intrusions, and lateral movement.

Architecture of RHACS

  • RHACS consists of two logical groups of services: Central and Secured Cluster services:
  • Central services act as a control plane. They provide a REST API, and a web console with vulnerability, compliance, policy, and violation management interfaces. The Central services also ensure data persistence and provide centralized image scanning.
  • Secured Cluster services provide components for monitoring workload definitions and pod activity, and enforcing security policies.

When registries that secured clusters use are not reachable from the control plane cluster, Secured Cluster services can also include image scanning.

Central Services Architecture
  • Central service has two components 
    1. central and its database
      • The database service that Central uses stores all configuration settings and runtime data on a persistent volume.
      • Central can validate workloads during deployment, and can monitor pods during operations, across the entire secured infrastructure. 
    2. scanner (stackrox scanner) and its database
      • The database service that Scanner uses stores image vulnerability data on an ephemeral volume for caching purposes.
      • The Scanner service collects configuration and software package information from every layer of every image that it is requested to scan.
    3. scanner V4
      • Scanner V4. The later scanner version consists of three components: Indexer, Matcher, and a database service. The database service in Scanner V4 uses a persistent volume. Scanner V4 adds support for new package databases and better vulnerability recognition to the existing StackRox Scanner service.
Secured Cluster Services Architecture
  • RHACS Secured Cluster services consist of three required components: Sensor, Admission Controller, and Collector.

In secured clusters that use registries that are not accessible to the central Scanner, you must also deploy the Scanner component. One such example is when the internal OpenShift image registry in a secured cluster is not exposed for external access.
