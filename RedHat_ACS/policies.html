Default Security Policies

  • The default policy configuration is a baseline posture for detecting the most common risks in your environment.
  • When customizing default policies, it is a good practice to first clone the existing policy, apply any customizations to the new clone, and disable the original default policy.
  • Each policy falls into one of the categories that RHACS organizes by type and function.

Enforcing a Deploy-time Policy

  • RHACS works with admission plug-ins in openshfit  to enforce policies before the their persistent in the ETCD and after the authentication and authorization layer.
  • RHACS uses the ValidatingAdmissionWebhook controller to verify that the resource that is being provisioned complies with the specified security policies. RHACS creates a ValidatingWebhookConfiguration object, which contains multiple webhook rules.
  • When the Kubernetes or OpenShift API server receives a request that matches one of the webhook rules, the API server sends an AdmissionReview request to RHACS. Then, RHACS accepts or rejects the request, based on the configured security policies.
  • Complementing the admission controller, the Collector component analyzes and monitors container activity on the cluster nodes, including Linux kernel-level activity, container runtime activity, host network traffic, and file systems.

The Collector operates during run time, and the admission controller enforces policies during workload deployment.

apiVersion: platform.stackrox.io/v1alpha1
kind: SecuredCluster
metadata:
  name: managed-cluster
  namespace: stackrox
spec:
  admissionControl:
    listenOnCreates: true 1
    listenOnEvents: true 2
    listenOnUpdates: true 3

With deploy-time enforcement, RHACS works with Kubernetes admission controllers and OpenShift admission plug-ins. RHACS prevents OpenShift from creating or updating workloads, for example, deployments, daemon sets or jobs, that match the conditions of the policy. 

  • Hard enforcement: the OpenShift API server blocks all noncompliant deployments.
  • Soft enforcement: It executes the enforcement by scaling down replicas to zero for deployments that violate a policy, and prevents pods from being scheduled.

Runtime Policies

  • configure runtime policies to trigger policy violations based on the following events:
    • Deployments: RHACS triggers policy violations when event sources include process and network activity, such as executing a command in a pod and pod port forwarding.
    • Audit logs: RHACS triggers policy violations when event sources match Kubernetes audit log records.
  • When a runtime policy uses the Inform and enforce enforcement method, RHACS deletes the pods for which an event matches the criteria of the policy.
  • setting the scope of the policy enforcement to explicitly include or exclude deployments.
  • RHACS first discovers existing processes and creates a baseline. On locking the baseline, RHACS operates in a deny-all mode and allows only processes that are listed in the baseline to run.
