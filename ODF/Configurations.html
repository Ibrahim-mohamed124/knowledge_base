A volume is a directory that is accessible to the containers in a pod. The volume type determines how that directory was created, the medium that backs it, and its contents.

  • Kubernetes supports two categories of volume types:
    • Ephemeral volume types, which have the lifetime of a pod. The most common types are: emptyDir, configMap, and secret.
    • Persistent volume types, which exist beyond the lifetime of a pod and are implemented as plug-ins. They are determined by the storage back end and by the access modes that they support.

The most common types are: csi (Container Storage Interface), nfs (Network File System), fc (Fiber Channel), and local (local storage devices that are mounted on nodes).

Lifecycle of PVs and PVCs

  • A control loop in the control plane is responsible for binding the PVCs to the PVs. The loop watches for new PVCs and finds any matching PV, and binds them together. If a PV was dynamically provisioned for a new PVC, then the loop always binds that PV to the PVC. The binding is a one-to-one mapping between the PV and the PVC.
  • The Delete reclaim policy removes the PV object from Kubernetes and removes the data from the storage back end. The Retain policy does not remove the data from the storage back end, even if you remove the PV manually.

 

The block storage type does not support RWX, and the volumes are Non-shareable: only one client at a time can mount an endpoint of this type.

    • ocs-storagecluster-ceph-rbd. Installed by the Rook-Ceph operator, it provides block storage, and uses Ceph RADOS Block Device (RBD) technology. It introduces the cephblockpool CR to enable creating and customizing storage pools.

OpenShift Data Foundation does not protect you from data corruption or from sharing files that should not be shared.  Additionally, for file storage data, the operating system and the file system handle the locking and caching of the files.

Using block storage is ideal for applications that require faster access for optimizing computationally heavy data workloads. Applications that use block-level storage implementations gain efficiencies by communicating at the raw device level, instead of relying on the OS layer access.

  • CSI driver primarily involves volume provisioning, attachment, and detachment not runtime I/O operations.
  • If a CSI Node Plugin DaemonSet faces issues like ImagePullBackOff, the Pod can continue to use the attached volume as long as it remains on the same node.

Crucial Note: The "Data Plane" vs. "Control Plane"

It’s a common misconception that data flows through the CSI driver. In reality:

Control Plane: CSI Driver handles attaching and mounting.

Data Plane: The Node's Operating System Kernel handles the actual movement of data between the app and the storage backend.

Normally, for a filesystem volume, the CSI driver would run mkfs to format the disk and then mount it to a directory.

For Block Storage: The CSI driver skips the formatting. It does not create an ext4 or xfs filesystem.

3. Device Mapping (The "Plug-in")

  • Instead of mounting a directory, Kubernetes creates a symbolic link (symlink) to the raw device inside the Pod's infrastructure.
    • On the Node, the device exists at a path like /dev/xvda.
    • Inside the Pod, that device is mapped to a path you specify (e.g., /dev/block-storage).

Configuring Applications to Use NFS Storage

  • within the RHOCP cluster, an application writes any data to the NFS share with the namespace UID and GID. From outside the cluster, the data is written with the UID of the external machine.
  • OpenShift Data Foundation uses CephFS internally to manage NFS storage.
  • By default, OpenShift Data Foundation does not provide the NFS storage class. You can enable the NFS feature when you create the storage cluster
apiVersion: ocs.openshift.io/v1
kind: StorageCluster
...output omitted...
spec:
...output omitted...
  nfs:
    enable: true
...output omitted..

 

oc patch storageclusters ocs-storagecluster   --namespace openshift-storage   --type merge --patch '{"spec": {"nfs":{"enable": true}}}'

 

  • After you enable the NFS feature, wait until the ocs-storagecluster-cephnfs resource is ready , and both  csi-nfsplugin and NFS server pods are created.
oc describe cephnfs ocs-storagecluster-cephnfs   -n openshift-storage
oc get pods -n openshift-storage -l app=csi-nfsplugin
oc get pods -n openshift-storage -l ceph_daemon_type=nfs
oc get sc
ocs-storagecluster-ceph-nfs      openshift-storage.nfs.csi.ceph.com            ...

Expose the NFS PV

  • configure a load balancer (LB) service such as MetalLB in your cluster to allow external access to the NFS server pod.
  • Get the value for the ceph_nfs field from the CephNFS pod that Rook deploys in the openshift-storage namespace:
oc get pod -l app=rook-ceph-nfs -L ceph_nfs

 

  • Expose the NFS service outside the cluster by creating an LB service.
apiVersion: v1
kind: Service
metadata:
  name: cephnfs-lb 
  namespace: openshift-storage
spec:
  ports:
    - name: nfs
      port: 2049
  type: LoadBalancer 
  externalTrafficPolicy: Local
  selector: 
    app: rook-ceph-nfs
    ceph_nfs: ocs-storagecluster-cephnfs

 

  • Get the name of the PV that is associated with the NFS PVC:
oc get pvc pvc_name --output jsonpath='{.spec.volumeName}{"\n"}'

 

  • Use the NFS PV name to get the NFS share path:
 oc get pv pv_name   --output jsonpath='{.spec.csi.volumeAttributes.share}{"\n"}'

 

  • Get the ingress IP for the NFS server from the LB service:
oc -n openshift-storage get service cephnfs-lb   --output jsonpath='{.status.loadBalancer.ingress[].ip}{"\n"}'

Defining Custom Storage Classes

  • The storage class objects are globally scoped, which means that only users with the cluster-admin or storage-admin roles can create them.
  • For defining the storage pool for a storage class, you can either use an existing storage pool or create another storage pool. Pools are logical partitions that store objects within Ceph.
    • The Rook operator enables creating and customizing storage pools through the CephBlockPool custom resource (CR).
apiVersion: ceph.rook.io/v1
kind: CephBlockPool
metadata:
  name: replicapool
  namespace: rook-ceph
spec:
  failureDomain: host 1
  replicated:
    size: 3 2
  deviceClass: hdd

 

Introducing Cross Storage Class Clones

  • OpenShift Data Foundation enables moving volumes from a storage class to another storage class that uses the same provisioner but different settings, such as the storage pool,the number of replicas, or the compression mode.
  • A clone is a duplicate of an existing storage volume that is used as any standard volume. You create a clone of a volume to make a point in time copy of the data. The back-end device creates an exact duplicate of the specified volume.
    • When creating a PVC clone, you must meet the following requirements:
    • Cloning support is only available for Container Storage Interface (CSI) drivers.
    • Cloning support is only available for dynamic provisioners.
    • The capacity of the PVC clone must be the same or larger than the capacity of the source volume.
    • Cloning can only be performed between two volumes that use the same volumeMode setting.
    • You can only clone a PVC when it exists in the same namespace as the destination PVC.
    • You can create up to 512 clones per PVC for both CephFS and RADOS Block Device.
    • You cannot clone a PVC from one provisioner to another provisioner, for example a PVC from RBD to CephFS.
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: test-clone
  namespace: dummy
spec:
  accessModes:
  - ReadWriteOnce
  dataSource: 1
    kind: PersistentVolumeClaim
    name: test
  resources:
    requests:
      storage: 2Gi
  storageClassName: pg-development-storage
  volumeMode: Filesystem

  • Single replica pools require a dedicated set of Ceph OSDs. If one of the OSDs fails, then the data is lost.
