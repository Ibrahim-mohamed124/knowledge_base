A volume is a directory that is accessible to the containers in a pod. The volume type determines how that directory was created, the medium that backs it, and its contents.

  • Kubernetes supports two categories of volume types:
    • Ephemeral volume types, which have the lifetime of a pod. The most common types are: emptyDir, configMap, and secret.
    • Persistent volume types, which exist beyond the lifetime of a pod and are implemented as plug-ins. They are determined by the storage back end and by the access modes that they support.

The most common types are: csi (Container Storage Interface), nfs (Network File System), fc (Fiber Channel), and local (local storage devices that are mounted on nodes).

Lifecycle of PVs and PVCs

  • A control loop in the control plane is responsible for binding the PVCs to the PVs. The loop watches for new PVCs and finds any matching PV, and binds them together. If a PV was dynamically provisioned for a new PVC, then the loop always binds that PV to the PVC. The binding is a one-to-one mapping between the PV and the PVC.
  • The Delete reclaim policy removes the PV object from Kubernetes and removes the data from the storage back end. The Retain policy does not remove the data from the storage back end, even if you remove the PV manually.

 

The block storage type does not support RWX, and the volumes are Non-shareable: only one client at a time can mount an endpoint of this type.

  • When deploying OpenShift Data Foundation by using local storage devices, you install the local storage operator as a prerequisite. The local storage operator creates a SC, commonly named lso-volumeset, to make disk devices available to the Rook-Ceph operator as PVs.
    • ocs-storagecluster-ceph-rbd. Installed by the Rook-Ceph operator, it provides block storage, and uses Ceph RADOS Block Device (RBD) technology. It introduces the cephblockpool CR to enable creating and customizing storage pools.
      • The SC uses the openshift-storage.rbd.csi.ceph.com provisioner, and its parameters define the file system type, Ceph pool, Ceph image format, and image features.
    • ocs-storagecluster-cephfs. Installed by the Rook-Ceph operator, this class provides shared and distributed file system data services, by using the Ceph File System (CephFS).
      • The SC uses the openshift-storage.cephfs.csi.ceph.com provisioner and it introduces the CephFilesystem CR that sets up and manages CephFS.
    • ocs-storagecluster-ceph-rgw. Installed by the Rook-Ceph operator, this class provides S3-compatible on-premise object storage, and uses the Ceph Object Gateway, which is also known as the radosgw RADOS Gateway (RGW) daemon, an HTTP server that interacts with the Ceph Storage Cluster.
      • This class introduces the cephobjectstore CR, which defines the settings and operational parameters for the RGW instance. The SC uses the openshift-storage.ceph.rook.io/bucket provisioner.
    • openshift-storage.noobaa.io. Installed by the NooBaa operator, this class uses NooBaa, which is a highly customizable and dynamic data gateway for objects, to provide the Multicloud Object Gateway (MCG) service.
      • This class introduces the NooBaa CR, which defines the configuration and deployment of the MCG service. This SC uses the openshift-storage.noobaa.io/obc provisioner.

OpenShift Data Foundation does not protect you from data corruption or from sharing files that should not be shared.  Additionally, for file storage data, the operating system and the file system handle the locking and caching of the files.

Using block storage is ideal for applications that require faster access for optimizing computationally heavy data workloads. Applications that use block-level storage implementations gain efficiencies by communicating at the raw device level, instead of relying on the OS layer access.
