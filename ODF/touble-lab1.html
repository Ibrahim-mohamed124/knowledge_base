• View the health status of the Ceph cluster by using the oc get command.
oc get cephcluster

 

  • View the cluster health details by using the oc describe command.
oc describe cephcluster | grep "^Status" -A15

 

Note

Ceph periodically checks the integrity of your data by scrubbing placement groups. A scrub error indicates that Ceph detects a consistency error in your data.

  • View the Ceph MON logs by using the oc logs command with the app=rook-ceph-mon label.
oc logs -l app=rook-ceph-mon --tail=-1 --since=1h \
  | grep ERR

 

  • View the Ceph OSD logs by specifying the app=rook-ceph-osd label for the oc logs command. Filter the log with the ERR flag.
oc logs -l app=rook-ceph-osd --tail=-1 --since=1h \
  | grep ERR

 

  • Create a Must-Gather archive
oc adm must-gather \
  --image=registry.ocp4.example.com:8443/odf4/odf-must-gather-rhel9:v4.16 \
  --dest-dir=mg
  • Obtain the cluster health with the ceph health command.
ceph  health detail

 

  • Use the rados list-inconsistent-obj command to get more information about the inconsistency.
 rados list-inconsistent-obj x.0 | jq .inconsistents[].shards

 

One or more causes can lead to PG inconsistencies. A PG repair can be dangerous in some cases and might result in a complete data loss.

Always consult with Red Hat support before attempting a PG repair.

Identify and Replace a Failing Drive and Node

  • View the health status of the Ceph cluster by using the oc get command.
[student@workstation ~]$ oc get cephcluster
  • enter the Ceph toolbox with the oc rsh command.
[student@workstation ~]$ oc rsh deploy/rook-ceph-tools
  • In the Ceph toolbox terminal, use the ceph status command to get more information about the storage cluster health.
sh-5.1$ ceph status
  cluster:
    id:     1b1e995b-f561-4699-b0f7-dbe012a15339
    health: HEALTH_WARN 
     2 osds down
 data:
   pgs:     100.000% pgs not active
  • List the Ceph OSD with the ceph osd tree command.
sh-5.1$ ceph osd tree
  • List the Ceph OSD deployment that is attached to the missing node.
[student@workstation ~]$ oc get deployments -l app=rook-ceph-osd \
  -l failure-domain=node_name
  • Scale down the Ceph OSD deployment from the previous step.
[student@workstation ~]$ oc scale deploy/rook-ceph-osd-0 --replicas 0 deployment.apps/rook-ceph-osd-0 scaled
  • Remove the Ceph OSD from the OpenShift Data Foundation cluster by using the ocs-osd-removal job template. Set the FAILED_OSD_IDS parameter to the value of the OSD ID from the previous steps.
[student@workstation ~]$ oc process ocs-osd-removal -p FAILED_OSD_IDS=0 \  -p FORCE_OSD_REMOVAL=true | oc create -f - job.batch/ocs-osd-removal-job created

 

OpenShift Data Foundation removes the OSD only if it can keep three replicas of the data in the cluster. On a cluster with three nodes, you must set the FORCE_OSD_REMOVAL parameter to true to remove an OSD.

  • Confirm that the job removes the Ceph OSD from the OpenShift Data Foundation configuration.
[student@workstation ~]$ oc logs -l job-name=ocs-osd-removal-job -f \  | grep -i "'completed removal'  cephosd: completed removal of OSD"
  • Clean up the ocs-osd-removal-job resource.
[student@workstation ~]$ oc delete job ocs-osd-removal-job job.batch "ocs-osd-removal-job"
  • In the Ceph toolbox terminal, confirm that the failing node and the associated Ceph OSD are removed from the configuration by using the ceph osd tree command.
sh-5.1$ ceph osd tree
Add the replacement node to the OpenShift cluster.
  • Configure the node storage with the local storage operator and add the node to the OpenShift Data Foundation cluster.
  • Confirm that the local volume operator creates another PV in the cluster.
[student@workstation ~]$ oc get pv
  • Add the cluster.ocs.openshift.io/openshift-storage label to the node to deploy the OpenShift Data Foundation components on that node.
[student@workstation ~]$ oc label no worker02 \  cluster.ocs.openshift.io/openshift-storage="" node/node_name labeled
  • Confirm that OpenShift schedules the OpenShift Data Foundation components on the node. Wait until both the OSD and MON pods are running.
[student@workstation ~]$ oc get po -owide | grep node_name
  • In the Ceph toolbox terminal, list the Ceph OSD with the ceph osd tree command. Confirm that both the new node and the associated Ceph OSD are in the OpenShift Data Foundation configuration.
sh-5.1$ ceph osd tree

 

Identify the remaining issue on the storage cluster.

  • Switch to the other terminal. Verify the status of the Ceph pods.
[student@workstation ~]$ oc get pods -o wide | grep -v Running
  • View the Ceph OSD logs by using the oc logs command with the pod name from the previous step.
[student@workstation ~]$ oc logs rook-ceph-osd-x

The Ceph OSD is failing due to a missing block device.

  • In the Ceph toolbox terminal, retrieve the device name that is attached to the faulty OSD by using the ceph device ls-by-daemon command. Use the OSD identifier from the previous steps.
sh-5.1$ ceph device ls-by-daemon osd.x
  • In the other terminal, connect to the compute node from the previous step by using the oc debug command and list the available disk devices in the /dev/ path.
[student@workstation ~]$ oc debug no/worker03 -- chroot /host ls -1 /dev/sd*

 

  • Remove the faulty OSD from OpenShift Data Foundation, by using the job template
  • In the Ceph toolbox terminal, list the Ceph OSD with the ceph osd tree command. Confirm that the node and the associated Ceph OSD are removed from the OpenShift Data Foundation configuration.
sh-5.1$ ceph osd tree
  • Switch to the other terminal. Verify that the Ceph OSD PV is in the released state. 
[student@workstation ~]$ oc get pv
  • Delete the persistent volume from the previous step.
[student@workstation ~]$ oc delete pv

Replace the faulty drive on the compute node. 

  • Wait until the local storage operator provisions the new disk.
[student@workstation ~]$ oc get pv
  • Wait until OpenShift Data Foundation creates the Ceph OSD pod.
[student@workstation ~]$ oc get pod -l app=rook-ceph-osd \  --field-selector spec.nodeName=node_name

In the Ceph toolbox terminal, verify the status of the storage cluster and monitor the recovery progress.
