Architecture of OpenShift Data Foundation

  • OpenShift Data Foundation provides access to file, block, and object storage, based on Ceph technology.
  • OpenShift Data Foundation uses the multicloud object gateway (MCG) service, which is based on the NooBaa project, to act as a data federation service. (The NooBaa operator manages MCG)
    • The MCG service provides a local object service (S3 API), which is backed by local storage or cloud-native storage.
  • Rook-Ceph storage operator to manage and orchestrate persistent volume claims and dynamic provisioning.
  • OpenShift Data Foundation can replicate and store data by using a controlled replication under scalable hashing (CRUSH) algorithm, which avoids a single point of failure.

Controlled Replication Under Scalable Hashing (CRUSH) is a pseudo-random, decentralized data distribution algorithm used in Ceph to determine object storage locations without relying on a central lookup table

OpenShift Data Foundation Architecture

  1. OpenShift Data Foundation: Acts as a meta-operator, and installs the Rook-Ceph and NooBaa operators as a requirement.
  2. OpenShift Container Storage: Provides the back-end services for file, block, and object data storage
  3. Multicloud Object Gateway: Provides the multicloud object gateway (MCG) service and the NooBaa cluster resource.

 

  • On-premise deployments use physical storage, which is passed to Ceph. Use this storage with RADOS gateway (RGW) class object bucket claims (OBCs) or MCG class OBCs.

radosgw is an HTTP REST gateway for the RADOS object store, a part of the Ceph distributed storage system. It is implemented as a FastCGI module using libfcgi, and can be used in conjunction with any FastCGI capable web server.

  • Cloud deployments use S3 storage, which the MCG service handles.
    • You can add S3 storage later, and handle it through the MCG service.
    • You can add Ceph RGW later.
  • MCG stand-alone deployments use any available PVCs, and handle them through the MCG service.
How the internal components work together?
  • Ceph services handle block storage within OpenShift Data Foundation.
    • The local storage operator manages the storage from the local disks, and the Ceph services handle the block storage volumes from the Ceph cluster.
    • The Ceph services can pass data to either the MCG service, for handling as object storage, or to a Kubernetes Container Storage Interface (CSI) plug-in for direct OS-level access by containers or virtual machines (VMs) - either block for file system storage -
    • The NooBaa and Rook-Ceph operators handle these OBCs and PVCs, respectively. The Rook-Ceph operator also handles RGW class OBCs.

Rook orchestrates the Ceph storage solution, with a specialized Kubernetes Operator to automate management. Rook ensures that Ceph will run well on Kubernetes and simplify the deployment and management experience.

  • MCG services handle cloud object storage within OpenShift Data Foundation, which is managed by NooBaa operator.
  • The Red Hat OpenShift Container Storage (OCS) operator provides additional internal management of the Rook-Ceph and the NooBaa operators.

OpenShift Data Foundation Deployment Modes

Internal Mode Deployment

  • Internal: All the OpenShift Data Foundation components run entirely within an RHOCP cluster. This mode uses dynamically provisioned persistent volumes (PVs) that the administrators specify in the installation wizard.
  • odf sizer
  • Types of internal deployment
    1. Simple: The simple deployment approach uses the local storage devices, portable storage devices, vSphere volumes, or storage area network (SAN) volumes. OpenShift Data Foundation services run jointly with applications.
    2. Optimized: The optimized deployment approach uses dedicated RHOCP infrastructure nodes for OpenShift Data Foundation.

External Mode Deployment

  • OpenShift Data Foundation uses independent, external Red Hat Ceph Storage services that run outside the cluster.

Deploying Multiple Storage Clusters

  • OpenShift Data Foundation supports deploying two storage clusters, where one is in internal mode and one is in external mode.
  • The first cluster must be installed in internal mode in the openshift-storage namespace, and the second cluster must be installed in external mode in the openshift-storage-extended namespace.

Disaster Recovery Architectures

  • OpenShift Data Foundation DR options provide volume-persistent data and metadata replication across sites that are geographically dispersed.
DR with stretch cluster
  • A single cluster is stretched across two locations to provide the storage infrastructure with DR capabilities, and a third location acts as the arbiter. 
  • OpenShift Data Foundation-based synchronous replication.
Regional-DR
  • Regional-DR is based on asynchronous volume level replication for block and file volumes, which provides protection against several failures with minimal potential data loss.
  • Regional-DR requires a hub cluster with the Red Hat Advanced Cluster Management for Kubernetes (RHACM) and MultiClusterHub operators, and primary and secondary managed clusters with OpenShift Data Foundation.
Metro-DR
  • OpenShift Data Foundation-based synchronous replication between two RHOCP clusters in two data centers.
  • Metro-DR requires an external RHCS cluster, two RHOCP clusters with OpenShift Data Foundation, and one hub cluster with RHACM to provide application and data mobility across RHOCP clusters.

 

Deploying OpenShift Data Foundation from the Web Console

  • The overall installation steps are as follows:
    1. Install the local storage operator (optional).
    2. Label at least three nodes that host the data services in the storage cluster (optional).
    3. Install the OpenShift Data Foundation operator (mandatory).
    4. Create the storage system (mandatory).
  • If using local volumes as storage, you must first install the local storage operator from the OperatorHub menu. The local storage operator creates the necessary Kubernetes custom resources (CRs) to enable the use of local volumes that are attached to the nodes.
    • The nodes must have at least one available empty raw block device.
  • Before installing the OpenShift Data Foundation operator, you must identify three nodes on which the OpenShift Data Foundation services are installed.
cluster.ocs.openshift.io/openshift-storage= label

Removing Storage System

  • Uninstall annotations:
uninstall.ocs.openshift.io/cleanup-policy

uninstall.ocs.openshift.io/mode
  • If the uninstall.ocs.openshift.io/cleanup-policy is set to delete, then the Rook-Ceph Storage Operator cleans up the physical drives and the DataDirHostPath. However, if the uninstall.ocs.openshift.io/cleanup-policy is set to retain, then it keeps the physical drives and DataDirHostPath.
  • If the uninstall.ocs.openshift.io/mode is set to graceful, then the Rook-Ceph and NooBaa operators pause the uninstallation process until the persistent volume claims (PVCs) and object bucket claims (OBCs) are removed manually. However, if the uninstall.ocs.openshift.io/mode is set to forced, then the Rook-Ceph and NooBaa operators proceed with the uninstallation process, even if the PVCs and the OBCs still exist.

The Rook-Ceph storage operator removes the PVCs and the NooBaa operator removes the OBCs.

oc annotate storagecluster ocs-storagecluster \
  uninstall.ocs.openshift.io/cleanup-policy="retain" --overwrite
oc annotate storagecluster ocs-storagecluster \
  uninstall.ocs.openshift.io/mode="forced" --overwrite
  • Uninstallation Prerequisites:
    • Ensure that the OpenShift Data Foundation cluster is in a healthy state. Otherwise, the process can fail because pods cannot be terminated successfully due to insufficient resources or other problems.
    • Ensure that applications or other services are not consuming PVCs or OBCs that use the storage classes from OpenShift Data Foundation.
    • After removing the dependent resources, delete any other CRs that were created by using the custom resource definitions from OpenShift Data Foundation.
    • Delete PVCs and OBCs that are provisioned by using OpenShift Data Foundation.
    • Delete the volume snapshots that are using OpenShift Data Foundation.
oc get volumesnapshot --all-namespaces
oc delete volumesnapshot VOLUME-SNAPSHOT-NAME -n NAMESPACE
