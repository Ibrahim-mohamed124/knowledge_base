Ceph Components

  • Ceph has a modular and distributed architecture with the following elements:
    1. An object storage back end, RADOS (Reliable Autonomic Distributed Object Store)
    2. Various access methods to interact with RADOS
  • RADOS is a self-healing and self-managing software-based object store.

The Ceph storage cluster has the following daemons:

  1. Monitors
    • Ceph Monitors (MONs) maintain maps of the cluster data and information about all other components.
    • Ceph MON is the brain of the storage cluster and helps the other daemons to coordinate with each other.
    • A cluster of Ceph MONs is spread across the storage nodes to ensure high availability if a node fails.

In Ceph’s decentralized architecture, MONs don't actually handle the storage of user data—that’s the job of the OSDs (Object Storage Daemons). Instead, MONs manage the Cluster Map, which is the master blueprint of the system's health and configuration.

  1. The Monitor Map: Tracks which MONs are currently active and their IP addresses.
  2. The OSD Map: Records which OSDs are "up" (running) and "in" (distributing data), along with their network locations.
  3. The PG Map: Tracks the status of Placement Groups (the logical "buckets" data lives in).
  4. The CRUSH Map: This is the "brain" of Ceph. It contains the physical hierarchy of your hardware (which OSDs are in which server, which servers are in which rack) and the rules for how data should be replicated.
  5. The MDS Map: Only used if you are using the Ceph File System (CephFS), tracking the Metadata Servers.
  1. Object Storage Daemons
    • Ceph Object Storage Daemons (OSDs) store data and handle data replication, recovery, and rebalancing. On a bare metal deployment, each OSD represents a dedicated disk on a compute node.
  2. Managers
    • Ceph Managers (MGRs) track runtime metrics and expose cluster information through a REST API. OpenShift then uses that API to show the storage cluster status and metrics on the web console.
  3. Metadata Servers
    • Ceph Metadata Servers (MDSes) store metadata that CephFS uses so that clients can run POSIX commands efficiently.
  4. RADOS Gateway
    • Ceph RADOS Gateway (RGW), which is also known as Ceph Object Gateway, provides an object storage interface that supports the Amazon S3 API.

Ceph in OpenShift Data Foundation

  • The Rook-Ceph operator manages the Ceph cluster. The operator creates and manages all the Ceph daemons as pods and uses the persistent volumes that the local storage operator creates to provision the Ceph OSDs.

 

  • In addition to Ceph daemons, the operator manages the CSI drivers for the storage classes. For each storage class that Ceph manages, two set of pods are deployed:
    1. The CSI plug-in pods run on all compute nodes and manage access to the Ceph volumes for applications that run on those nodes.
    2. The CSI provisioner pods monitor PVCs and create, resize, and handle snapshots for the matching volumes on the Ceph cluster.

Inspecting Ceph Components Logs

oc logs -n openshift-storage -l ceph_daemon_type=mon

 

  • Regardless of whether a logging stack is deployed on the OpenShift cluster, OpenShift Data Foundation stores the Ceph component logs on the storage nodes for persistence, in the /var/lib/rook/openshift-storage/log path.
 oc debug no/worker01 -- chroot /host \
  ls -al /var/lib/rook/openshift-storage/log/
  • Must Gather
 oc adm must-gather \
  --image=registry.redhat.io/odf4/odf-must-gather-rhel9:v4.16

By default, the oc adm must-gather command writes to a must-gather.local.identifier directory on your local system. To write to a specific local directory, you can also use the --dest-dir option.

Using the Ceph Toolbox

  • To use the Ceph CLI, you must first enable the Ceph toolbox. Modify the storage cluster resource with oc edit and add the enableCephTools setting as follows:
apiVersion: ocs.openshift.io/v1
kind: StorageCluster
metadata:
  name: ocs-storagecluster
  namespace: openshift-storage
...output omitted...
spec:
  enableCephTools: true
 oc -n openshift-storage rsh deployment/rook-ceph-tools
ceph health
ceph status
ceph df
ceph osd tree

Using the OpenShift Data Foundation CLI

  • The odf get health command verifies the health of the Ceph cluster and validates that OpenShift Data Foundation components are correctly scheduled and are running. here
odf get health
odf get rook status  ==  oc describe cephcluster

 

Identifying Disk Failure in OpenShift Data Foundation

  • disks should be replaced when they show slow I/O, crashing OSDs, or data corruption.
Slow I/O
  • OpenShift Data Foundation reports any Ceph OSD that cannot service I/O operations per second (IOPS) in the queue within a certain baseline. In such scenarios, the storage cluster fires a SLOW_OPS alert, and the ceph health detail command lists the impacted OSDs:
ceph health detail
Crashing OSDs
ceph status
 cluster:
    id:     6642...0c1b
    health: HEALTH_WARN
            20 daemons have recently crashed
  • Ceph OSD pod logs to get more information about the error
ceph crash ls-new or ls
  • OpenShift Data Foundation collects crash logs and stores them on the node in the /var/lib/rook/openshift-storage/crash/posted/<crash-id> path. 
  • ceph crash archive command archives a specific crash, and the ceph crash archive-all command archives all new crashes.
ceph crash archive 2024-07-10T20:34:31.245716Z...f366
