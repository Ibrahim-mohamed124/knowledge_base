Resource Quotas for Storage Resources

  • OpenShift provides two types of quotas to manage the resource consumption, one for single projects and another for multiple projects, To create a resource quota for multiple projects, define a shared cluster resource quota to restrict resource usage across more than one namespace.
  • By using the storage-class-name.storageclass.storage.k8s.io prefixes, you can specify the resource quota for a storage class. With this approach, specialized storage classes, such as those with faster throughput disks or larger storage pools, can have stricter constraints.
  • You can set resource quotas across projects to enforce a baseline storage allocation for the applications in each project.
apiVersion: quota.openshift.io/v1
kind: ClusterResourceQuota
metadata:
  name: for-user
spec:
  quota: 1
    hard:
      persistentvolumeclaims: "2"
  selector:
    annotations: 2
      openshift.io/requester: username
    labels: null 3

 

oc describe appliedclusterresourcequotas

Persistent Volume Claim Expansion

  • You can expand the following PVC types:
    • PVC with ReadWriteOnce or ReadWriteMany access that is based on Ceph File System with the Filesystem volume mode.
    • PVC with ReadWriteOnce access that is based on Ceph RADOS Block Devices with the Filesystem volume mode.
    • PVC with ReadWriteOnce access that is based on Ceph RADOS Block Devices with the Block volume mode.
    • PVC with ReadWriteOncePod that is based on Ceph File System or Network File System with the Filesystem volume mode.
    • PVC with ReadWriteOncePod access that is based on Ceph RADOS Block Devices with the Filesystem volume mode. With ReadWriteOncePod access mode, you mount the volume as read-write by a single pod on a single node.
  • PVC expansion does not support the following scenarios:
    • Recovering from failure when expanding volumes
    • PVC shrinking

Reclaim Space from the OpenShift Data Foundation Cluster

  • OpenShift does not automatically remove from the back-end storage the volumes and the snapshots with a Retain reclaim policy. To delete the data from these volumes and snapshots to free space, you must manually act in OpenShift Data Foundation.
    • Orphaned volumes can eventually fill the storage cluster, which leads to a storage-full scenario. Therefore, you should identify and delete volumes and snapshots from the Ceph cluster.

Remove Orphaned Volumes

  • If you delete a volume or a snapshot with a Retain reclaim policy from the OpenShift cluster, then the volume is deleted from the OpenShift cluster but not from the OpenShift Data Foundation cluster.
  • To identify orphaned volumes, you must compare the volumes in OpenShift with the volumes in OpenShift Data Foundation. If OpenShift Data Foundation has more volumes than OpenShift, then you have orphaned volumes that you must clean.
  • Reclaim space operations are supported only on Ceph RBD volumes.
  • You can use the following methods to reclaim the space from RBD volumes:
    1. Use a reclaim space job
    2. Use a reclaim space cron job
    3. Use the reclaim space annotation on the PVC (Recommended)
  • The reclaim space operation frees space from the Ceph cluster by executing the following commands:
    • fstrim, which OpenShift executes on file-system volumes that are mounted to a pod
    • rbd sparsify, which OpenShift executes on volumes that are not mounted to any pod
Reclaim Space by Using a Job
apiVersion: csiaddons.openshift.io/v1alpha1
kind: ReclaimSpaceJob
metadata:
  name: reclaim-space
spec:
  target:
    persistentVolumeClaim: data 1
  timeout: 360 2
  backOffLimit: 10 3
  retryDeadlineSeconds: 900 4
oc get reclaimspacejobs
Reclaim Space by Using a Cron Job
apiVersion: csiaddons.openshift.io/v1alpha1
kind: ReclaimSpaceCronJob
metadata:
  name: reclaim-space-schedule
spec:
  jobTemplate:
    spec:
      target:
        persistentVolumeClaim: data 1
      backOffLimit: 10 2
      retryDeadlineSeconds: 900 3
      timeout: 360 4
  schedule: '@weekly' 5
  concurrencyPolicy: Forbid 6
  failedJobsHistoryLimit: 1 7
  successfulJobsHistoryLimit: 3 8
 oc get reclaimspacecronjobs
Use the Reclaim Space Annotation
  • You can use the reclaimspace.csiaddons.openshift.io/schedule annotation to the PVC so that OpenShift reclaims space automatically based on a given schedule. This operation creates a ReclaimSpaceCronJob resource based on the schedule that you provide in the annotation.
 oc annotate pvc data \
  reclaimspace.csiaddons.openshift.io/schedule="0 0 * * 0"
  • You can also use the reclaimspace.csiaddons.openshift.io/schedule annotation to the namespace so that OpenShift reclaims space automatically based on a given schedule for all the PVCs that you create in the namespace.
oc annotate namespace test \
  reclaimspace.csiaddons.openshift.io/schedule=@weekly

Adding Disks to the OpenShift Data Foundation Cluster

  • For internal deployments that use local storage devices, OpenShift Data Foundation enables you to scale by adding disks to the storage nodes or by adding storage nodes.
    • Adding disks to the nodes
    • Adding nodes
    • Enabling any existing unused disks on the nodes

When enabling or adding unused disks to nodes, or adding new nodes, the new disks must be the same size and type as the disks that are used during the deployment, because OpenShift Data Foundation does not support heterogeneous disks.

You can use a maximum of nine storage devices per node. The high number of storage devices leads to a higher recovery time during the loss of a node.

Adding Nodes

  • To add the nodes to the OpenShift cluster, depending on the type of infrastructure, perform the following steps:
    • Get new machines with the required infrastructure.
    • Create OpenShift Container Platform compute nodes by using the new machines.
    • Check for certificate signing requests (CSRs) in Pending state.
    • Approve all the required CSRs for the new nodes.
  • apply the OpenShift Data Foundation label to the nodes with the worker label:
oc label nodes \
  -l node-role.kubernetes.io/worker= cluster.ocs.openshift.io/openshift-storage=
  • The LocalVolumeDiscovery object and the LocalVolumeSet object use the OpenShift Data Foundation label in the nodeSelector field. The local volume discovery automatically detects the disks in the new nodes, and the local volume set automatically provisions the disks on the new nodes.
---
apiVersion: local.storage.openshift.io/v1alpha1
kind: LocalVolumeDiscovery
metadata:
  name: auto-discover-devices
  namespace: openshift-local-storage
spec:
  nodeSelector:
    nodeSelectorTerms:
    - matchExpressions:
      - key: cluster.ocs.openshift.io/openshift-storage
        operator: Exists
...output omitted...
---
apiVersion: local.storage.openshift.io/v1alpha1
kind: LocalVolumeSet
metadata:
  name: lso-volumeset
  namespace: openshift-local-storage
spec:
...output omitted...
  nodeSelector:
    nodeSelectorTerms:
    - matchExpressions:
      - key: cluster.ocs.openshift.io/openshift-storage

  • inspect the LocalVolumeSet object to verify the number of provisioned devices.
oc get localvolumesets/lso-volumeset \
  -o custom-columns='"NAME:metadata.name,STORAGECLASS:spec.storageClassName,\
  PROVISIONED:status.totalProvisionedDeviceCount,VOLUME_MODE:spec.volumeMode"' \
  -n openshift-local-storage
