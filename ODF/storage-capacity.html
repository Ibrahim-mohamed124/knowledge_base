Resource Quotas for Storage Resources

  • OpenShift provides two types of quotas to manage the resource consumption, one for single projects and another for multiple projects, To create a resource quota for multiple projects, define a shared cluster resource quota to restrict resource usage across more than one namespace.
  • By using the storage-class-name.storageclass.storage.k8s.io prefixes, you can specify the resource quota for a storage class. With this approach, specialized storage classes, such as those with faster throughput disks or larger storage pools, can have stricter constraints.
  • You can set resource quotas across projects to enforce a baseline storage allocation for the applications in each project.
apiVersion: quota.openshift.io/v1
kind: ClusterResourceQuota
metadata:
  name: for-user
spec:
  quota: 1
    hard:
      persistentvolumeclaims: "2"
  selector:
    annotations: 2
      openshift.io/requester: username
    labels: null 3

 

oc describe appliedclusterresourcequotas

Persistent Volume Claim Expansion

  • You can expand the following PVC types:
    • PVC with ReadWriteOnce or ReadWriteMany access that is based on Ceph File System with the Filesystem volume mode.
    • PVC with ReadWriteOnce access that is based on Ceph RADOS Block Devices with the Filesystem volume mode.
    • PVC with ReadWriteOnce access that is based on Ceph RADOS Block Devices with the Block volume mode.
    • PVC with ReadWriteOncePod that is based on Ceph File System or Network File System with the Filesystem volume mode.
    • PVC with ReadWriteOncePod access that is based on Ceph RADOS Block Devices with the Filesystem volume mode. With ReadWriteOncePod access mode, you mount the volume as read-write by a single pod on a single node.
  • PVC expansion does not support the following scenarios:
    • Recovering from failure when expanding volumes
    • PVC shrinking

Reclaim Space from the OpenShift Data Foundation Cluster

  • OpenShift does not automatically remove from the back-end storage the volumes and the snapshots with a Retain reclaim policy. To delete the data from these volumes and snapshots to free space, you must manually act in OpenShift Data Foundation.
    • Orphaned volumes can eventually fill the storage cluster, which leads to a storage-full scenario. Therefore, you should identify and delete volumes and snapshots from the Ceph cluster.

Remove Orphaned Volumes

  • If you delete a volume or a snapshot with a Retain reclaim policy from the OpenShift cluster, then the volume is deleted from the OpenShift cluster but not from the OpenShift Data Foundation cluster.
  • To identify orphaned volumes, you must compare the volumes in OpenShift with the volumes in OpenShift Data Foundation. If OpenShift Data Foundation has more volumes than OpenShift, then you have orphaned volumes that you must clean.
  • Reclaim space operations are supported only on Ceph RBD volumes.
  • You can use the following methods to reclaim the space from RBD volumes:
    1. Use a reclaim space job
    2. Use a reclaim space cron job
    3. Use the reclaim space annotation on the PVC (Recommended)
