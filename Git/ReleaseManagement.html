Release Management

  • Release management is the process of planning, coordinating, and controlling how software changes move from development into production safely and predictably.
    • In this process you have to balance speed with stability!!!

Release Types 

1. Scheduled Releases

  • Fixed release windows (e.g., bi-weekly, monthly)
  • Common in enterprises and regulated environments

2. On-Demand Releases

  • Released when features are ready
  • Enabled by CI/CD and automation

3. Emergency / Hotfix Releases

  • Minimal scope
  • Bypass some gates but still auditable

Release Gates (Quality Controls):

Common gates:

  • Automated tests
  • Code quality (SonarQube)
  • Security scans
  • Manual approval (CAB, product owner)

Versioning Strategies

Semantic Versioning 

  • Format: MAJOR.MINOR.PATCH
    • MAJOR – Breaking changes
    • MINOR – Backward-compatible features
    • PATCH – Backward-compatible fixes

Environment Promotion Strategies

  • Core Principle:  Build once, promote many times.
  • You never rebuild the artifact per environment.
  • Typical Environment Flow
    • DEV → QA / SIT → UAT → STAGING → PROD
  • Each promotion:
    • Uses the same artifact
    • Adds approvals and validations
    • Changes configuration, not code

Promotion Models

1. Artifact Promotion (Best Practice)

  • Artifact stored once (Artifactory)
  • Promoted via metadata or repo move
  • Guarantees immutability
  • Example:
    • docker-dev → docker-prod
    • libs-snapshot → libs-release

2. Pipeline-Driven Promotion

  • Same artifact reference passed between pipeline stages
  • Promotion controlled by pipeline logic
  • Used when:
    • Artifactory promotion features are unavailable
    • Simpler setups

3. GitOps Promotion (Modern)

  • Artifact version referenced in Git
  • Promotion = Git commit/PR
  • Argo CD / Flux syncs environment
  • Example:
    • Update Helm values with new image tag
    • PR approval = promotion
