service monitors 
  • High level abstraction CRD to add scrape configs to promethues configuration 
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  labels:
    release: prometheus
    app: prometheus
  name: api-service-monitor
  namespace: api
spec:
  jobLabel: job
  endpoints:
  - interval: 30s
    port: web #name
    path: /swagger-stats/metrics

  selector:
    matchLabels:
      app: service-api
  • endpoints: provide the scrape_configs data
  • selector: to target specific services
  • jobLabels: to choose  a custom label from the service  to put its value in the job label in  Prometheus
  • The  release: prometheus label tell the prometheus instance to use this service  monitor # this label must exist
k get prometheuses.monitoring.coreos.com -o yaml
    serviceMonitorNamespaceSelector: {}
    serviceMonitorSelector:
      matchLabels:
        release: prometheus

 

prometheusrules
  • To add rules to prometheus
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  labels:
    release: prometheus
  name: api-rules
  namespace: default #by default prometheus operator only picks prometheusrules in the default namespace but it can be modified in the values.yaml file
spec:
  groups:
  - name: api
    rules:
    - alert: down
      annotations:
        description: {{$labels.instance}/{{$labels.pod}} 
      expr: up == 0
      labels:
        severity: critical

Thanos
  • Allows to pull metrics (query) from many promethues servers with deduplications.
  •  Store the metrics in a long term storage
  • Thanos is deployed as a side car container along with the promethues container. And it is responsible for receiving the metrics from promethues and pushing them to the long term storage. 
  • Thanos store responsible for pulling the metrics from the long term storage.
  • Thanos  Query servers the metrics to endpoints like Grafana and clients from different locations
  • Disable promethues compaction  by setting these two values to the same value, as thanos is not compatible with compaction 
--storage.tsdb.min-block-duration 2hr
--storage.tsdb.max-block-duration 2hr
Scaling and HA
  • Vertical scaling: Increase the number of promethues instances, and each instance is responsible for a set of applications

 

  • Horizontal scaling: If one service has many instances, split the service instance into sets, which will be monitored with different promethues instances (horizontal sharding)
  • To limit the number of samples per a scrape
scrape_configs:
  - job_name:
    static_configs:
    - targets:
    sample_limit: 1000
Prometheus Federation
  • One global promethues server acts as a proxy for other promethues servers that collect metrics from different sites, leading to one UI with the metrics from all sites
scrape_configs:
- job_name: "federation"
  honor_labels: true # to keep the original promethues labels
  metrics_path: "/federate" # every promethues server exposes the metrics it collected in this endpoint
  params:
    match[]:
    - 'job="only the jobs needed to be collected centrally"'
  static_configs:
  - targets: [promtheues endpoints list:9090]
