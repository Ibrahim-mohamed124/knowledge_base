Alert Manager:

  • Prometheus generates alerts based on conditions written in PromQL, then alert manager takes these alerts and routes them to a notification mechanism.
  • Alerting rules are used by promethues to write the conditions, on which Prometheus will generate alerts
groups:
  - name: node
    interval: 15s
    rules:
     - record: ...
     - alert: LowMemory
       expr: node_memory_memFree_percent < 20

 

  • To tell promethues to evaluate the condition for some time before firing the alert use the for attribute
groups:
  - name: node
    interval: 15s
    rules:
     - alert: NodeDown
       expr: up{job="nodes"} == 0
  • Alert states:
    • Inactive: the condition is false
    • Pending: the condition is true but waiting for the for period to end
    • Firing: condition is true
  • To add a label to an alert
groups:
  - name: node
    interval: 15s
    rules:
     - record: ...
     - alert: NodeDown
       expr: up{job="node"} == 0
       for: 10m
       labels:
         severity: high
       annotations:
         description: "only for describtion that are able to use go templating language"

 

  • To access the labels in annotations
{{.Labels}} > to get access to alert labels
{{.Lables.instance}}  > to get acess to instance labels
{{.Value}} > to get the metric value when the alert is firing

Alert Manager architecture:

  • One alert manager can be used with many promethues servers
  • Dispatching: groups the alerts together
  • inhibition: suppress alerts if other alerts exist
  • silencing: silence alerts by silence rules
  • routing:  what alert get sent to who ?
  • notification engine: send alerts

Installation:

  • Get the binray
wget https://github.com/prometheus/alertmanager/releases/download/v0.31.1/alertmanager-0.31.1.linux-amd64.tar.gz
tar -xvaf alertmanager-0.31.1.linux-amd64.tar.gz

 

  • configure promethues to use alert manager:
global:
alerting:
 alertmanagers:
 - static_configs:
   - targets:
     - alertmanager_IP:9093
 - "rules.yml"

Installation using systemd
  • put the binaries to  /usr/local/bin and create alertmanager user
useradd --no-create-home --shell /bin/false alertmanager
mkdir /etc/alertmanager
mv alertmanager.yml /etc/alertmanager/
chown -R alertmanager:alertmanager /etc/alertmanager
mkdir /var/lib/alertmanager 
chown -R alertmanager:alertmanager /var/lib/alertmanager > to store the data
mv alertmanager amtool /usr/local/bin/
chown alertmanager:alertmanager  /usr/local/bin/alertmanager  /usr/local/bin/amtool

 

[Unit]
Description=Alertmanager
Wants=network-online.target
After=network-online.target

[Service]
User=alertmanager
Group=alertmanager
Type=simple
ExecStart=/usr/local/bin/alertmanager\
   --config.file /etc/alertmanager/alertmanager.yml \
   --storage.path /var/lib/alretmanager/
Restart=always

[Install]
WantedBy=multi-user.target
Configurations
  • There are main three sections
    • global: applies to all sections
    • route: routing
    • receivers: who?

Route:

  • receiver: global receiver
  • group_by:  grouping
  • routes: override the global receiver and group_by
    • match_re: like matchers but use regex
    • matchers: the labels based on which the routing will happen
    • receiver: the specific receiver to this labels
    • subroutes: to take an alert from the group and send it to different notifier
    • continue: true to send the same alert to more than one receiver
Receivers and Notifiers
  • Receivers: responsible for taking a group of alerts and making notifications
  • Notifiers: responsible for sending these notifications to targets
route:
 receiver: team
global:
  smtp_from: source@mail
receivers: 
 - name: team
   slack_configs:
   - api_url: "url"
     channel: "channel"
   email_configs:
   - to: target@mail
     from: source@mail
     smarthost: mail_server
     auth_username: source@mail
     auth_identity: source@mail
     auth_password: "password"
  • Global configurations: to config all the receivers
  • To customize the notification use Go templating language with:
    • GroupLabels > contains the group labels of the nofitication
    • CommonLabels > labels that are common across all alerts in the notification
    • CommonAnnotations 
    • ExternalURL > URL of the Alertmanager
    • Status > will be set to firing if at least one alert in the notification is firing
    • resolved > if all alerts are resolved
    • Receiver > name of the receiver
    • Alerts > List of all the alerts in the notification
      • Labels > labels of the alert
      • Annotations 
      • status
      • StartsAt
      • EndsAt
