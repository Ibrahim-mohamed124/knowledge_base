Intro to Observability

  • The ability to understand and measure the state of a system based upon data generated by the system
  • It allows to generate actionable outputs from unexpected scenarios in dynamic environments and help in
    • Better insights into the internal working of a system
    • speed up troubleshooting
    • Monitor performance
  • Unlike monolithic application microservices applications are much more complex
  • it gives us answers for
    • error rates
    • high latency
    • timing outs

Pillars of Observability

  1. logging:
    • Records of events that have occurred and encapsulated information about the specific event with timestamps
  2. metrics:
    • numeric data about the system that contain:
      • Metric name
      • Value of the metric
      • Timestamp
      • Dimensions: more data about the mertic
  3. Traces:
    • allow to follow operations as they traverses through various systems & services hop by hop to understand the whole picture
      • trace ID: the unique id for a trace
      • Spans: evens in the trace and contain start time of the span, duration, and parent id

Quality of Service indicators

  1. measurable targets/goals to help organizations strike the right balance between product development and operations work.
    • SLI: service level indicator is a quantitative measure of some aspect of the level of service that is provided like request latency, error rate, saturation, throughput, and availability.
      • Metrics that accurately measure the user's experience.
    • SLO: service level objective - target value or range  of an SLI
    • SLA: service level agreement:  Contract between end user and provider to meet the SLO

Prometheus

Use cases

  • collect metrics from many datacenters
  • alerting when conditions are met
  • scrape and do operations in metrics.
  • data charting

Basics

  • Prometheus collects metrics by scraping targets who expose metrics through an HTTP endpoint
  • Store the scraped metrics into time based database
Architecture
  1. Data retrieval workers:  scrape the data from the target endpoints by sending http requests ( pulling metrics )
  2. TSDB:  stores the metric data
  3. HTTP server: accepts PromeQL Query
  4. Exporters: runs on the targets so data retrieval can scrape the metrics
  5. Push Gateway: for short lived jobs that cannot wait till the data retrievals pull the metrics so the gateway pushes stores the metrics till the data retrieval workers pull the data
  6. Service Discovery: Dynamically update the list of target to monitor
  7. Alert manager: to send alerts

 

Collecting Metrics
  • Prometheus collects metrics by sending http requests to /metrics endpoint of each target by default
  • If the system has no metrics endpoint use exporter process to collect metrics and exposing /metrics endpoint
  • client libraries allow to expose any application metrics for promethues for different languages like go and python

 

In Prometheus version 3.5.0 and above, the consoles/ and console_libraries/ directories have been removed from the distribution tarball.

Installation as systemd services

  • Install the binary from the official docs
wget https://github.com/prometheus/prometheus/releases/download/v3.9.1/prometheus-3.9.1.linux-amd64.tar.gz

 

  • unarchive the tar file
 tar -xvaf prometheus-3.9.1.linux-amd64.tar.gz

 

  • create a user for prometheus without a login shell
useradd -no-create-home --shell /bin/false prometheus

 

  • make the configuration and data directories
mkdir /etc/prometheus && mkdir /var/lib/prometheus
chown prometheus:prometheus /etc/prometheus && chown prometheus:promethues /var/lib/promethues

 

  • move prometheus and promtool binaries  to /usr/local/bin/ and change ownership
mv prometheus promtool /usr/local/bin/ && chown prometheus:prometheus /usr/local/bin/prometheus /usr/local/bin/promtool

 

  • move prometheus.yaml to /etc/prometheus/ and change ownership
  • create the service file
[Unit]
Description=Prometheus
Wants=network-online.target
After=network-online.target

[Service]
User=prometheus
Group=prometheus
Type=simple
ExecStart=/usr/local/bin/prometheus \
   --config.file /etc/prometheus/prometheus.yml \
   --storage.tsdb.path /var/lib/prometheus/     \

[Install]
WantedBy=multi-user.target

 

  • copy the file to the /etc/systemd/system/promethues.service and enable the service

Metrics

  • the metric has standard structure
<metric_name>{comma seperated list of labels}<value><timestap>

 *  The labels are the instances of a metric like a class and instance of that class in programming

 * Every metrics has two labels by default (instance, job)

the metric name is save under the __name__ label inside the prometheus server

 

  • the metrics are stored with timestamp using the Unix Epoch time
  • promethues stores these metrics with unique set of labels and name in a time series
  • Every metric has two attributes
    1. Type:  specify the type of that metric
    2. Help: description of what is that metric
Types of metrics
  • Counter
    • how many times an event happened like total number of  http requests
  • Gauge
    • measure the current value of event, can go up or down like current CPU utilization
  • Histogram
    • groups  metrics for in certain limits(quantiles), how many total events in a period of time like request size and response time
  • Summary
    • like histogram but does not need quantiles ahead of time like percentages of response time for a number of requests

PromeQL

  • query language in promethues
PromQL data types
  1. String > simple string value
  2. scalar > numeric floating value
  3. Instant Vector > a set of time series with a single sample for each time series, all sharing the same timestamp
    • like querying node-cpu-seconds-total will return a number of timeseries' (metric + unique labels) for each cpu. all the returned values share the same timestamp
  4. Range Vector > set of time series with a range of different data points over time for each time series
    • like querying node-cpu-seconds-total{selectors}[xm]. The data that will be returned will cover a range of x mintues

In Prometheus (specifically within PromQL), the difference between a scalar and a vector lies in whether the data represents a single numerical value or a set of time-series data with labels. 

Here is a breakdown of the differences based on Prometheus Query Language (PromQL) data types:

1. Scalar (Single Value)

 

  • Definition: A simple, constant numeric floating-point value.
  • Characteristics: It has no time series data associated with it, no timestamps, and no labels (tags).
  • Examples: 10, 1.5, 3.14159, 86400.
  • Usage: Used as parameters in functions (e.g., topk(3, ...)), or in mathematical calculations against vectors (e.g., http_requests_total * 2).

2. Vector (Time-Series Set)

 

  • Definition: A collection of time series, where each time series usually has a single sample (value) associated with a timestamp.
  • Characteristics: Includes labels (e.g., instance="node1", job="api") which allow for filtering and grouping.
  • Types:
    • Instant Vector: A set of time series containing a single sample for each, all sharing the same timestamp. Example: node_cpu_seconds_total.
    • Range Vector: A set of time series containing a range of data points over time (e.g., node_cpu_seconds_total[5m]).
  • Usage: The main data retrieved from Prometheus to create graphs, alerts, and perform calculations.
Selector & Matchers
  • to select only a subset of timeseries' of a metric
  • types of matchers
    • equality =  > node-filesystem-avail-bytes{instance="node1"}
    • negative equality ≠ node-filesystem-avail-bytes{instance="node1, device!≠"tmpfs"}
    • regular expression =~ node-filesystem-avail-bytes{device=~"/dev/sda.*}
    • negative regex !~ node-filesystem-avail-bytes{mountpoint!~"/boot".*}
offset modifier 
  • instead of getting the latest value, get a value from the history
  • metric{selector} offset xU >> unit cloud be days, weeks, hours, minutes
  • to go back to specific point in time use @ with unix timestamp
node_memory_Active_bytes{instance="10.200.130.134:9100"}[2m] @1770360941 offset 10m
Operators
  • Arithmetic operators
metric (+|*|-|/)

 

  • Comparison operators
metric (<|>|=)

 

  • bool operator returns true or false
metric (<|>|=) bool value
  • Logical operators
metric AND metric OR|UNLESS

 

  • Vector Matching: operations on the metrics against each others
    • samples with the exactly the same labels get matched together
    • All labels must be same for samples to match
    • To match samples with different labels use the ignore keyword
metric / igonring(lables) metric1

      * To match on specific label use the on(label) keyword

metric / on(labels) metric1

 

  • Types of matching
    • one to one > one sample and another
    • many to one >  many samples for one
      • group_left key tells  the server that the many is on the lift
      • group_right key tells the server that the many is on the right
  • Aggregators
    • aggregate vectors(metrics) values [like functions]
      • sum(), max(), avg()
    • to aggregate vectors by labels use the by
sum by(labels) (metric)

    * to ignore labels in the aggregation use the without keyword

sum without(labels) (metric)

 

  • Functions:
    • math floor(metrics), ceil()
    • time()
    • sort()
    • changing types of data from scalar to vector(value in a time series)
    • absent(), absent-over-time() and present(), present-over-time()
    • rate(),irate()
Subqueries 
  • Subqueries in Prometheus PromQL allow executing a range query within an instant query, acting as an in-line, ad-hoc recording rule to analyze historical data trends. They enable applying range-vector functions (like rate or max_over_time) over a historical period, returning a new range vector, typically used for complex, long-term trend analysis or alert calculations.
<instant_query> [<range>]  [offset: duration]

 

  • range: the data grouping period (like for rate function)
  • offset: query range (get data from last x minutes)
  • duration: query step for subquery
max_over_time(rate(prometheus_http_requests_total[1m]) [5m:30s])

Histogram

  • There are three sub metrics in histogram metric
    • Count: total number of samples
    • Sum: sum of all the samples
    • Bucket: a count of samples with the a label (le) all these samples have values that are less than or equal to value of that label
      • These buckets are cumulative( include all the pervious )
rate(requests_latency_seconds_count[1m]) [5m: 30s]
rate(requests_latency_seconds_bucket[1m]) [5m: 30s]
rate(requests_latency_seconds_sum[1m])
rate(requests_latency_seconds_sum[5m]) / rate(requests_latency_seconds_count[5m])

 * le label cloud be le="+Inf" > all the samples

rate(request_latency_seconds_bucket{path="/articles", le="0.06"}[1m]) \ ignoring(le) 
rate(request_latency_seconds_bucket_count{path="/articles"}[1m])
Quantiles
  • Determine the count of  values in a distribution are above or below a certain limit by percentage
  • Use the histogram_quantile() function to easily calculate quantile values
histogram_quantile(value, metreic)

 

  • Quantiles can be used to ensure that SLO are met.
  • This function uses linear interpolation to estimate the value at a specific percentage of samples to get the actual number of samples equal or less that specific value make sure that you have a bucket that matches that value.
  • Adding more buckets at the quantile range makes it more accurate, but more buckets more RAM and disk space usage.

Summary  metric

  • has three submetrics like the histogram (count, sum, and quantiles instead of buckets)
  • useful to get the percentage values instead of just numbers of samples

Recording Rules

  • An expression that is run continuously and the results are stored in a time series to be used by other tools
  • The rules are in rules.yml file  and restarting the promethues server
global:
  rules_files:
  - rules.yml

 

  • rules file:
groups: >> group of rules that are run sequentially 
  - name: group_name
    interval: evaluation_interval
    rules:
     - record: rule_name
       expr: promql_expr
       labels: 
         label: value >> can be used to group alerts
     - record: rule_name
       expr: promql_expr
       labels: 
         lable: value
  - name: group_name
  ....

 

  • groups run parallel but rules inside each group run sequentially
  • best practices in naming the recording rules
    • level:metric_name:operations
      • level: indicates the aggregation level of the metric y the labels is has, and always includes the job label + others
      • Metric_name
      • Operations:  list of aggregators or functions that applied on the metric values
    • examples:
      • job-method-path:http_errors:rate5m
groups:
 - name: example1
   intrval: 15s
   rules: 
    - record: "job:node_filesystem_free_percent:avg"
      expr: avg by(instance) (node_filesystem_free_percent)

 

  • We can avoid updating the prometheus.yml file every time for adding a new rule file we create by making use of globs.
rule_files:
  - "*rules.yaml"
